# 哒咔 Android 应用需求文档

## 1. 产品概述

哒咔是一款每日打卡类日记应用。应用每天为用户推荐一个可执行的小事项，用户完成后可以通过文字、图片、视频等方式记录当天发生的事情，并在之后按日期、缩略图或推荐流的形式回顾、查看和编辑记录。

首版数据保存在本地。设计和实现需要预留后续在线化能力，避免从纯本地应用升级到账号、云同步、推荐服务时发生大规模重构。

## 2. 产品目标

- 降低日记记录门槛，通过每日推荐事项触发用户记录。
- 支持文字、图片、视频等多媒体记录方式。
- 支持历史记录浏览、查看、编辑，并保留每一次编辑版本。
- 支持推荐事项按关键词分类，并配有对应图片。
- 使用 Jetpack Compose 和 Material Design 构建一致、现代的 Android UI。

## 3. 目标用户

- 希望记录日常生活但缺少固定主题的人。
- 希望通过轻量任务改善生活、娱乐、工作节奏的人。
- 希望回顾过往记录、查看自己状态变化的人。

## 4. 核心用户流程

### 4.1 每日推荐打卡流程

1. 用户打开应用。
2. 首页展示当天推荐事项。
3. 用户点击推荐事项的操作入口。
4. 进入记录编辑页面。
5. 用户填写发生了什么，并可添加图片、视频等媒体。
6. 用户保存记录。
7. 记录写入本地数据库，并进入历史记录列表。

### 4.2 历史记录浏览流程

1. 用户进入历史记录页面。
2. 页面按日期列出已有记录，并展示缩略图、标题、分类、摘要等信息。
3. 用户可切换到类似走马灯的推荐浏览模式。
4. 用户点击某条记录。
5. 进入记录详情页，可查看完整内容和媒体。
6. 用户点击编辑，修改记录内容。
7. 保存后生成新的编辑版本，旧版本保留。

### 4.3 个人配置流程

1. 用户进入个人页面。
2. 查看或配置通用项目。
3. 配置项保存在本地，后续可映射到云端用户配置。

## 5. 功能需求

### 5.1 首页

首页是应用启动后的默认页面。

必须包含：

- 当日推荐事项卡片。
- 推荐事项标题。
- 推荐事项分类关键词，例如美食、运动、通勤、娱乐、工作、学习、社交、家务、放松。
- 推荐事项配图。
- 推荐文案或一句简短说明。
- “去记录”操作入口。
- 今日是否已记录的状态。

推荐事项示例：

- 美食：吃一份黄焖鸡米饭。
- 通勤：换一种交通工具出行。
- 运动：饭后散步 20 分钟。
- 工作：整理一次桌面或待办列表。
- 娱乐：看一集没看过的纪录片。
- 社交：给一位朋友发一条问候。
- 放松：睡前听 10 分钟轻音乐。

交互要求：

- 点击推荐事项或“去记录”进入记录编辑页面。
- 如果当天已有记录，入口可显示“查看今日记录”或“继续编辑”。
- 推荐事项应每日固定，避免同一天多次打开应用时随机变化。

### 5.2 记录编辑页面

用于创建或编辑打卡日记。

必须包含：

- 推荐事项信息展示。
- 日期。
- 文本输入区域，用于记录“发生了什么”。
- 图片添加入口。
- 视频添加入口。
- 已添加媒体列表或缩略图。
- 保存按钮。
- 取消或返回操作。

建议包含：

- 标题字段，可自动使用推荐事项标题，也允许用户修改。
- 心情或状态字段。
- 地点字段，首版可手动输入，后续可接入定位。
- 标签字段。

媒体要求：

- 支持从相册选择图片。
- 支持从相册选择视频。
- 首版可仅保存本地文件 URI 或应用私有目录路径。
- 删除媒体时只解除记录关联；是否删除本地文件需按实现策略明确。

编辑版本要求：

- 每次保存编辑都生成一个版本。
- 版本记录至少包含版本号、编辑时间、文本内容、媒体关联快照、标题、标签等。
- 详情页可查看当前版本。
- 后续需要提供版本历史入口，用户可以查看每一版本的编辑内容。

### 5.3 历史记录页面

用于浏览之前的打卡日记。

必须包含：

- 按日期倒序展示记录。
- 每条记录展示日期、标题、分类、摘要、缩略图。
- 点击记录进入详情页。
- 支持空状态。

建议浏览方式：

- 日期列表模式：按天展示，适合查找。
- 缩略图网格模式：突出图片和视频。
- 走马灯推荐模式：横向或全屏卡片轮播，用于随机回顾或推荐浏览。

筛选和排序建议：

- 按分类关键词筛选。
- 按是否包含图片/视频筛选。
- 按时间范围筛选。

### 5.4 记录详情页面

用于查看单条记录的完整内容。

必须包含：

- 标题。
- 日期和时间。
- 推荐事项分类。
- 正文内容。
- 图片和视频展示。
- 编辑入口。
- 版本历史入口。

编辑要求：

- 点击编辑进入记录编辑页面。
- 保存编辑后生成新版本。
- 旧版本不可被覆盖。

### 5.5 版本历史页面

用于查看记录每一次编辑。

必须包含：

- 当前记录的版本列表。
- 每个版本展示版本号、编辑时间、摘要。
- 点击版本可查看该版本完整内容。

建议包含：

- 标识当前版本。
- 对比两个版本的基础能力，首版可暂不实现。
- 恢复到历史版本的能力，首版可作为预留。

### 5.6 个人页面

用于配置通用项目。

首版建议配置项：

- 推荐分类偏好。
- 默认媒体保存策略。
- 是否显示每日提醒。
- 提醒时间。
- 主题模式：跟随系统、浅色、深色。
- 数据管理：导出本地数据、清理缓存。
- 关于应用。

后续在线化预留：

- 登录状态。
- 用户资料。
- 云同步状态。
- 备份与恢复。

## 6. 推荐事项需求

### 6.1 推荐事项字段

推荐事项至少包含：

- id。
- title：标题。
- description：简短说明。
- category：分类关键词。
- imageUri 或 imageAsset：配图。
- enabled：是否启用。
- createdAt。
- updatedAt。

### 6.2 分类关键词

首版推荐分类：

- 美食。
- 运动。
- 通勤。
- 娱乐。
- 工作。
- 学习。
- 社交。
- 家务。
- 放松。
- 探索。

### 6.3 推荐策略

首版本地推荐：

- 内置一批推荐事项。
- 每天根据日期稳定选择一个推荐事项。
- 避免短期内重复推荐同一事项。
- 可结合用户偏好过滤分类。

后续在线推荐：

- 推荐事项来源可替换为远程接口。
- 本地保留缓存。
- 推荐结果需要支持离线可用。

## 7. 数据需求

### 7.1 本地存储

首版数据保存在本地，建议使用 Room 数据库管理结构化数据，媒体文件保存在应用私有目录或通过系统媒体访问框架维护引用。

建议实体：

- CheckInRecord：打卡记录。
- CheckInVersion：记录版本。
- MediaAttachment：媒体附件。
- Recommendation：推荐事项。
- UserPreference：用户配置。

### 7.2 数据模型建议

CheckInRecord：

- id。
- recommendationId。
- title。
- date。
- category。
- currentVersionId。
- createdAt。
- updatedAt。
- deletedAt。

CheckInVersion：

- id。
- recordId。
- versionNumber。
- titleSnapshot。
- contentSnapshot。
- categorySnapshot。
- editedAt。

MediaAttachment：

- id。
- recordId。
- versionId。
- type：image 或 video。
- uri。
- thumbnailUri。
- sortOrder。
- createdAt。

Recommendation：

- id。
- title。
- description。
- category。
- imageUri。
- source：local 或 remote。
- enabled。

UserPreference：

- id。
- key。
- value。
- updatedAt。

### 7.3 在线化预留

为避免后续重构，首版应保持以下结构：

- Repository 层不直接暴露 Room DAO。
- 数据源拆分为 LocalDataSource 和 RemoteDataSource。
- UseCase 或业务服务层负责推荐、记录保存、版本生成等业务规则。
- 数据实体和 UI 状态分离，避免 UI 直接依赖数据库实体。
- 所有核心数据使用稳定 id，后续可映射服务端 id。
- 数据表预留 syncStatus、remoteId、updatedAt 等字段或设计迁移空间。

## 8. 技术约束

- Android 原生应用。
- UI 使用 Jetpack Compose。
- 设计遵循 Material Design。
- 建议使用单 Activity + Compose Navigation。
- 本地数据库建议使用 Room。
- 用户配置建议使用 DataStore。
- 媒体加载建议使用 Compose 兼容图片加载库。
- 视频播放可使用 AndroidX Media3。
- 依赖注入可使用 Hilt 或 Koin，按项目实际技术栈决定。

## 9. 页面结构建议

底部导航建议包含：

- 今日。
- 记录。
- 我的。

页面路由建议：

- HomeScreen：首页。
- EditRecordScreen：创建/编辑记录。
- HistoryScreen：历史记录。
- RecordDetailScreen：记录详情。
- VersionHistoryScreen：版本历史。
- ProfileScreen：个人页面。

## 10. UI/UX 要求

- 遵循 Material Design 组件规范。
- 使用 Material 主题色、排版、间距和状态反馈。
- 支持浅色和深色模式。
- 页面层级清晰，首页突出“今日推荐”和“去记录”。
- 历史页强调可浏览性，缩略图尺寸稳定，避免列表跳动。
- 图片和视频需要有加载、失败、空状态。
- 保存、删除、返回等操作需要明确反馈。
- 重要操作避免误触，例如删除记录、删除媒体。

## 11. 非功能需求

- 本地数据读写应可靠，保存失败需要提示用户。
- 历史列表滚动应流畅。
- 媒体缩略图应缓存，避免频繁加载原始视频或大图。
- 应支持 Android 常见屏幕尺寸。
- 应处理无媒体权限、文件不存在、视频不可播放等异常状态。
- 本地数据后续需要支持导出或备份。

## 12. 首版范围

首版必须完成：

- 首页每日推荐。
- 创建记录。
- 支持文字、图片、视频。
- 历史记录列表。
- 记录详情。
- 编辑记录并保存版本。
- 个人配置基础项。
- 本地数据保存。

首版可暂缓：

- 账号登录。
- 云同步。
- 版本差异对比。
- 历史版本恢复。
- 智能推荐算法。
- 社交分享。

## 13. 待确认问题

- 第 5 条“其他需求”当前为空，需要补充或确认删除。
- 是否需要每日提醒通知。
- 图片和视频是否需要应用内拍摄能力，还是只从相册选择。
- 是否需要密码、指纹或隐私保护。
- 是否需要数据导出格式，例如 JSON、Markdown 或 ZIP。
- 推荐事项配图是内置本地资源，还是允许后续从远程下载。
